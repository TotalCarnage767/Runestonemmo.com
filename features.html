RunestoneCore

The Engine That Powers Every System, Every Module, Every Adventure.

RunestoneCore is the foundational module of the RunestoneMMO ecosystem — a modular, extensible, MMO-grade engine providing the essential infrastructure: economy, auctions, shops, moderation, scoreboards, cooldowns, attributes, world rules, daily rewards, combat scaffolding, storage, and APIs. It boots through a deterministic CoreBootstrap sequence, ensuring each subsystem is initialized in order.

Core Systems

Configuration & Lifecycle

  • Multi-file YAML configuration (config, economy, combat, attributes, daily rewards, cooldowns, moderation, messages, playerdata, mob health, scoreboards, menus, teleport, resources)
  • Routed config access (economy, auction, shops, treasury, etc.) via ConfigManager
  • Subsystem lifecycle with CoreSubsystem and SubsystemManager (init + shutdown)
  • Task and listener registries to prevent orphaned schedulers and duplicate listeners
  • Adventure-based messaging with MiniMessage formatting and category prefixes

Player Data & Profiles

  • CorePlayer model with join info, playtime, titles/tags, toggles, preferences, attributes map
  • Dedicated combat XP and progression state
  • Wallet and bank balances across multiple currencies
  • Daily rewards state (cycle day, streaks, completion)
  • Async load/save with merge logic and caching
  • YAML storage backend (config-driven; database backends planned)

Economy Platform

  • Multi-currency registry with primary and secondary currencies
  • Wallet economy with events, logging, and per-player currency files
  • Bank accounts with interest, tax integration, and commands
  • Tax rules for payments, bank interest, auctions, and shops
  • Currency exchange with configurable rates and fees
  • Server treasury for routed taxes and admin sinks/sources
  • Dynamic pricing framework (config-gated)
  • Ledger and transaction logging for auditability
  • Vault bridge to expose balances to external plugins

Auctions & Player Shops

  • GUI auction house with escrow and expiration tasks
  • Listing validation, fees, cooldowns, and offline delivery queue
  • Player shops with listings, stock, taxes, and persistence
  • Shop creation and editor flows (including description support)
  • Integrated with economy, taxes, ledger, and logging

Daily Rewards

  • Calendar-based daily reward cycles with streaks and streak weeks
  • Timezone-aware eligibility checks
  • Per-day rewards (currency, items, commands)
  • Optional loot-table-style weekly/daily reward routing (config-driven)
  • /daily claim and /daily status commands

Combat & Attributes

  • Combat XP system with XP curves and level-up events
  • Damage engine with attack, defense, toughness, crit chance/damage, and biome multipliers
  • Minimum damage floors, particles, and sounds for feedback
  • Mob attribute application on spawn from combat and attribute configs
  • Attribute manager for per-entity stats and scaled player attributes
  • Hooks for future buffs/debuffs and temporary attributes

Scoreboards & Menus

  • Config-driven scoreboards with multiple layouts and conditions (combat, menus, worlds, regions, permissions, thresholds)
  • Optional override of external scoreboard plugins
  • Menu framework for building GUI menus with cleanup and lifecycle awareness

Resources & HUD

  • Resource pools (mana/stamina) with listener hooks for costs and regeneration
  • Resource HUD support (bossbar mode) for clean, compact display
  • Teleport cost integration (mana costs via configurable rules)

Teleport & Travel

  • TPA system with request/accept/deny flow and safety checks
  • Teleport toggle support (/tptoggle)
  • Homes system (persistent homes configuration and manager)
  • Back command support for returning to previous locations
  • Spawn system support for returning players to the realm gate

Cooldowns & Mob Health

  • Global and per-player cooldown registry
  • Centralized cleanup via task registry
  • Mob health bars using bossbars with configurable styles and caps
  • Per-player scan caps for performance safety

Items, Drops & VillageCraft

  • Config-driven item definitions with usage gating hooks
  • Drop tables subsystem for controllable rewards
  • VillageCraft hooks for server crafting extensions
  • Holograms support (config-driven)

Moderation & World Rules

  • Punishments (ban, mute, warn, etc.) with storage and events
  • Reports, staff notes, player state tracking, and cleanup tasks
  • Staff chat, chat control (lock, slowmode, filters), and channel routing
  • Jail system with locations, escape radius, and retention
  • Vanish, god mode, freeze, teleport, and inventory view tools
  • Maintenance mode, audit trail, and staff alerts
  • World rules for gameplay toggles, safe zones, and restricted worlds
  • Sleep system with one-player or percentage-based night skipping

Tasks, Logging & APIs

  • Central task registry and listener registry
  • Scheduled tasks for bank interest, auction expiration, scoreboards, moderation cleanup, and more
  • Economy and moderation logging, plus audit trail for staff actions
  • Public APIs: RunestoneCoreAPI, RunestoneEconomyAPI, event bus, and more

RPG Systems Not Included (Yet)

These systems are planned for future modules and are not part of RunestoneCore itself:

XP & Progression

  • No unified XP service for all skills (RunestoneXPService is defined but not implemented)
  • No global leveling system for non-combat skills
  • No skill trees or point allocation
  • No class or specialization framework

Content Systems

  • No quest framework
  • No achievements or title unlock systems beyond basic titles/tags
  • No unified loot table engine shared across modules
  • No custom item stat system or rarity tiers

Social & Villages

  • No Villages system (planned as the primary social grouping module)
  • No town/village claiming, protections, or treasuries in Core
  • No social ranks or village diplomacy in Core

Persistence & Scaling

  • Cooldowns and economy data persist via disk-backed storage
  • Attributes are primarily runtime-driven and configurable; broader persistence is planned
  • No MySQL/PostgreSQL storage implementation
  • No multi-server sync or proxy-aware scaling layer
